================================================================================
  MOODIFY — DOCUMENTACIÓN COMPLETA DE ARQUITECTURA Y ESTRUCTURA
  Fecha: Mayo 2026
================================================================================

  Moodify es una app de bienestar mental que combina registro de estados de ánimo,
  seguimiento de hábitos, chat con IA (psicóloga "Bloom"), mensajería P2P entre
  usuarios y una comunidad/foro social.

  Tecnologías principales:
  ─ Frontend:  React Native + Expo (Expo Router, TypeScript)
  ─ Backend:   Laravel 13 + Sanctum (API REST), SQLite
  ─ IA:        Ollama (modelo qwen3-vl:30b) via PsicologoService
  ─ Auth:      Bearer tokens (Laravel Sanctum)

  2. FRONTEND — PANTALLAS (RUTAS EXPO ROUTER)
================================================================================

  Expo Router usa el sistema de archivos para definir rutas, igual que Next.js.
  Cada archivo en /src/app/ es una ruta navegable.

  ── / (index.tsx) ─────────────────────────────────────────────────────────────
  Pantalla de bienvenida. Muestra logo, tagline y botones para Login/Registro.
  Estado local: showLogin (bool), showRegister (bool)
  Renderiza: <Login /> o <CreateUser /> según el estado.
  Tras login exitoso → router.replace('/dashboard')

  ── /dashboard (dashboard.tsx) ────────────────────────────────────────────────
  Pantalla principal tras iniciar sesión.
  Renderiza: <DashboardView />
  Consume: UserContext (accessToken, user)
  Contiene: saludo, racha, alerta de ánimo, estadísticas, cita del día.

  ── /calendar (calendar.tsx) ──────────────────────────────────────────────────
  Calendario mensual de estados de ánimo.
  Renderiza: <CalendarView />
  Consume: UserContext
  Contiene: grid de emojis por día, registro de estado, resumen mensual.

  ── /community (community.tsx) ────────────────────────────────────────────────
  Feed de comunidad y foro social.
  Renderiza: <ForumView />
  Consume: UserContext
  Contiene: tabs Feed/Personas, posts, likes, sugerencias de amigos.

  ── /profile (profile.tsx) ────────────────────────────────────────────────────
  Perfil completo y centro de ajustes del usuario.
  Consume: UserContext (userValue, setUserValue, logout)
  Contiene:
    ─ Hero card: avatar, nombre, email, puntos, racha
    ─ Tab "Perfil": HabitProgress, ChallengesSection, AchievementsBar
    ─ Tab "Ajustes": cuenta, privacidad, idioma, notificaciones, logout
    ─ Modal: Editar perfil (nombre, username, email)
    ─ Modal: Cambiar contraseña (con verificación de fortaleza)
    ─ Modal: Selector de idioma
  API calls:
    ─ POST /update-profile
    ─ POST /change-password
    ─ POST /update-settings

  ── /settings (settings.tsx) ──────────────────────────────────────────────────
  Redirige a /profile. Mantiene compatibilidad con enlace desde el navBar.

  ── /chat (chat/index.tsx) ────────────────────────────────────────────────────
  Lista de conversaciones activas (P2P y con Bloom).
  Consume: UserContext
  API calls: GET /conversations
  Muestra: avatar, nombre, último mensaje, hora, badge de no leídos.
  Contiene: botón para buscar usuarios y abrir chat P2P.

  ── /chat/@username (chat/[@username].tsx) ────────────────────────────────────
  Chat individual entre dos usuarios.
  Param: username del destinatario.
  API calls:
    ─ GET /messages/user/{recipientId} (carga historial)
    ─ POST /messages (envía mensaje)
    ─ POST /messages/read/{senderId} (marca como leídos)
  Muestra: burbujas de mensajes, estado online, personalización de tema.

  ── /chat/bloom (chat/bloom.tsx) ──────────────────────────────────────────────
  Chat con la psicóloga IA "Bloom".
  API calls: POST /chat (cada mensaje)
  Muestra: mensajes de usuario y respuestas de Bloom.
  El historial se mantiene en la conversación con type='ai'.

  ── /challenge/[id] (challenge/[id].tsx) ──────────────────────────────────────
  Vista de detalle de un reto específico.
  Param: id del reto.
  Renderiza información del reto y estado de progreso.

  ── /explore (explore.tsx) ────────────────────────────────────────────────────
  Sección informativa/educativa. Actualmente placeholder.

  ── AUTENTICACIÓN ─────────────────────────────────────────────────────────────

  Login.jsx
  ─ Props: ninguno
  ─ Consume: UserContext (setUserValue), useRouter
  ─ Campos: emailUsername, password, showPassword (toggle)
  ─ API: POST /api/login → { success, access_token, user }
  ─ On success: setUserValue({ accessToken: data.access_token, user: data.user })
  ─ Navega a: /dashboard

  CreateUser.jsx
  ─ Props: ninguno
  ─ Consume: ninguno (solo notificación visual)
  ─ Campos: fullName, username, email, password, confirmPassword
  ─ API: POST /api/register → { status, user }
  ─ On success: muestra toast de éxito, no redirige (usuario debe iniciar sesión)

  ── PROVIDERS ─────────────────────────────────────────────────────────────────

  user-provider.tsx (UserContext)
  ─ Estado: userValue = { accessToken: string|null, user: object|null }
  ─ Expone: userValue, setUserValue, logout
  ─ logout(): setUserValue({ accessToken: null, user: null }) + router.replace('/')
  ─ Helper: getUserThemeFromContext(userValue) → { bgName, myMsgColor, otherMsgColor, textColorOther, textColorOwn }
  ─ Usado en: TODOS los componentes que necesitan auth o datos del usuario

  ── NAVEGACIÓN ────────────────────────────────────────────────────────────────

  ─ API: POST /api/saveSleepLog → { date, start_time, end_time, total_minutes }

  Stress.tsx (modal)
  ─ Test de 10 preguntas tipo Likert (escala 1-5) para medir estrés percibido
  ─ API: POST /api/store → { total_score, breakdown }

  4. FRONTEND — USERCONTEXT Y FLUJO DE AUTENTICACIÓN
================================================================================

  ESTRUCTURA DE USERCONTEXT
  ──────────────────────────
  userValue: {
    accessToken: string | null,     // Bearer token de Sanctum
    user: {
      id:                  number,
      name:                string,
      email:               string,
      username:            string,
      type_user:           'admin' | 'user',
      points:              number,
      streak:              number,
      last_streak_day:     string | null,  // ISO datetime
      last_seen_at:        string | null,
      image_id:            string | null,
      theme_color:         string,         // '#6366F1' por defecto
      bg_image:            string | null,
      my_msg_color:        string,         // '#6366F1' por defecto
      other_msg_color:     string,         // '#F1F5F9' por defecto
      text_colorOwn:       string,         // '#FFFFFF' por defecto
      text_colorOther:     string,         // '#FFFFFF' por defecto
      is_public:           boolean,        // true por defecto
      show_in_community:   boolean,        // true por defecto
      notifications_enabled: boolean,     // true por defecto
      app_bg:              string,         // 'default' por defecto
      language:            string,         // 'es' por defecto
    } | null
  }

  FLUJO DE AUTENTICACIÓN
  ──────────────────────
  1. REGISTRO (CreateUser.jsx)
     ─ POST /api/register { fullName, username, email, password }
     ─ Backend: crea usuario + auto-verifica email (email_verified_at = now())
     ─ Respuesta: { status: 'success', user }
     ─ Frontend: muestra éxito, NO redirige. Usuario debe iniciar sesión.

  2. LOGIN (Login.jsx)
     ─ POST /api/login { emailUsername, password }
     ─ Backend: acepta email O username, valida contraseña, comprueba verificación
     ─ Respuesta: { success: true, access_token, user, token_type: 'Bearer' }
     ─ Frontend: setUserValue({ accessToken: data.access_token, user: data.user })
     ─ Navega a: /dashboard

  3. USO DEL TOKEN
     ─ Todos los requests autenticados incluyen: Authorization: Bearer {accessToken}
     ─ El token se almacena solo en memoria (UserContext) — se pierde al cerrar la app
     ─ No hay persistencia local (AsyncStorage) — hay que hacer login cada vez

  4. ACTUALIZAR CONTEXTO (patchContext pattern)
     ─ Cuando el backend devuelve un user actualizado:
       setUserValue((prev) => ({ ...prev, user: updatedUser }))
     ─ Esto provoca re-render de todos los componentes que consumen UserContext

  5. LOGOUT (profile.tsx)
     ─ Alert de confirmación
     ─ logout() → setUserValue({ accessToken: null, user: null }) + router.replace('/')

  CÓMO SE USA EL TOKEN EN CADA COMPONENTE
  ────────────────────────────────────────
  const { userValue } = useContext(UserContext);
  const token = userValue?.accessToken;

  const authHeaders = {
    Authorization: `Bearer ${token}`,
    Accept: 'application/json',
    'Content-Type': 'application/json',
  };

  // Cada fetch autenticado:
  const r = await fetch(`${API}ruta`, { headers: authHeaders });

================================================================================
  6. BACKEND — RUTAS API (routes/api.php)
================================================================================

  BASE URL: http://localhost:8000/api/

  ── SIN AUTENTICACIÓN ─────────────────────────────────────────────────────────

  POST   /login              AuthController@login
  POST   /register            AuthController@register

  ── CON AUTENTICACIÓN (middleware auth:sanctum) ───────────────────────────────

  GET    /user                → Devuelve usuario autenticado (Sanctum helper)

  ── RACHA Y ALERTAS ───────────────────────────────────────────────────────────

  ── AuthController.php ────────────────────────────────────────────────────────

  login(Request $request)
  ─ Valida: emailUsername (required), password (required)
  ─ Detecta si es email o username con filter_var(FILTER_VALIDATE_EMAIL)
  ─ Busca usuario en BD, verifica contraseña con Hash::check
  ─ Comprueba email_verified_at no sea null
  ─ Genera token: $user->createToken('auth_token')->plainTextToken
  ─ Retorna: { success: true, access_token, user, token_type: 'Bearer' }
  ─ Errores: 401 credenciales incorrectas, 403 email no verificado

  register(Request $request)
  ─ Valida: fullName, username (unique), email (unique), password (min:8)
  ─ Crea usuario con User::create([name, email, username, password hash])
  ─ Auto-verifica: $register->email_verified_at = now(); $register->save();
  ─ Retorna 201: { status: 'success', message, user }

  ── UserController.php ────────────────────────────────────────────────────────

  streakRegister(Request $request)
  ─ Obtiene usuario autenticado
  ─ Si never registered: inicia racha, +3 puntos
  ─ Si registered today: devuelve estado actual (no modifica)
  ─ Si yesterday: incrementa racha +1, +3 puntos
  ─ Si días perdidos: puede recuperar a coste de 6 puntos/día perdido
  ─ Actualiza: streak, points, last_streak_day
  ─ Retorna: { ok, streak, points, days_missed, recovery_cost }

  actionAlert(Request $request)
  ─ Comprueba si existe mood_register con user_id + date = today
  ─ Retorna: { ok: true, exists: bool, user: object }

  saveSleepLog(Request $request)
  ─ Valida: date, start_time, end_time, total_minutes
  ─ updateOrCreate([user_id, date], [start_time, end_time, total_minutes])
  ─ Retorna: { ok: true, data: sleepLog }

  fillInHours()
  ─ Busca sleep_log de hoy para el usuario autenticado
  ─ Retorna: { ok: true, data: sleepLog | null }

  updateProfile(Request $request)
  ─ Valida: name (required), username (unique excepto propio), email (unique excepto propio)
  ─ $user->update([name, username, email])
  ─ Retorna: { status: 'success', user: object completo }

  changePassword(Request $request)
  ─ Valida: current_password (required), new_password (min:8)
  ─ Verifica contraseña actual con Hash::check
  ─ Actualiza: $user->update(['password' => Hash::make($new_password)])
  ─ Retorna: { status: 'success' } | 422 si contraseña incorrecta

  updateSettings(Request $request)
  ─ Valida (todos nullable): is_public, show_in_community, notifications_enabled, app_bg, language
  ─ $user->update($request->only([campos]))
  ─ Retorna: { status: 'success', user: object completo }

  updateTheme(Request $request)
  ─ Acepta: bgName, myMsgColor, otherMsgColor, textColorOwn, textColorOther
  ─ Mapea a campos de BD: bg_image, my_msg_color, other_msg_color, etc.
  ─ Retorna: { status: 'success', user: object con tema actualizado }

  ── MoodRegisterController.php ────────────────────────────────────────────────

  saveMood(Request $request)
  ─ Valida: mood (required, enum de ~28 valores), daily_text?, date?
  ─ Crea nuevo registro (permite múltiples por día → historial)
  ─ Retorna: { ok: true, data: { id, user_id, mood, daily_text, date } }

  getMoodCalendar(Request $request)
  ─ Obtiene todos los registros del mes actual del usuario
  ─ Agrupa por fecha, toma el ÚLTIMO de cada día
  ─ Retorna: { data: { "2026-05-10": "happy", "2026-05-09": "calm", ... } }

  getTodayTimeline(Request $request)
  ─ Obtiene todos los registros de hoy ordenados por fecha desc
  ─ Retorna: [{ id, mood, time: ISO_string, text: daily_text }]

  getMonthlyStats(Request $request)
  ─ Agrupa registros del mes por mood con COUNT
  ─ Retorna: [{ mood: 'happy', total: 8 }, { mood: 'calm', total: 5 }, ...]

  getDashboardInfo(Request $request)
  ─ Consulta en paralelo: último mood, sleep de hoy, racha actual
  ─ Retorna: { lastMood: 'happy'|null, sleepMinutes: int|0, streak: int }

  ── ChatController.php ────────────────────────────────────────────────────────

  index() → GET /conversations
  ─ Busca conversaciones donde user_id = auth OR recipient_id = auth
  ─ Solo type = 'p2p'
  ─ Append para cada conversación: last_message, last_message_sender_id, last_message_at, unread_count
  ─ Retorna: array de conversaciones con datos de destinatario

  getMessagesByUserId($recipientId) → GET /messages/user/{recipientId}
  ─ Busca o crea conversación P2P entre los dos usuarios
  ─ Carga historial de messages_p2p ordenado por created_at asc
  ─ Actualiza last_seen_at del usuario autenticado
  ─ Retorna: { messages[], recipient_online: bool (last_seen < 5min), conversation }

  storeP2P(Request) → POST /messages
  ─ Valida: mensaje (required), recipient_id (required)
  ─ Crea o reutiliza conversación P2P
  ─ Inserta en messages_p2p: { conversations_id, sender_id, content }
  ─ Retorna: { status: 'success', conversation_id, message: objeto }

  markAsRead($senderId) → POST /messages/read/{senderId}
  ─ Actualiza read_at = now() para todos los mensajes de senderId en la conversación
  ─ Retorna: { status: 'success' }

  searchUsers(Request) → GET /users/search?q=
  ─ Si q vacío: retorna []
  ─ WHERE username LIKE '%{q}%' AND id != auth_user
  ─ Para cada resultado: comprueba si auth_user sigue al usuario (is_following)
  ─ Retorna: [{ id, name, username, image_id, streak, is_following }]

  toggleFollow(Request) → POST /users/follow
  ─ Valida: followed_id (required)
  ─ Comprueba en followed_follower si ya existe el par (follower_id, followed_id)
  ─ Si existe: DELETE (dejar de seguir) → retorna { status: 'unfollowed' }
  ─ Si no existe: INSERT → retorna { status: 'followed' }

  store(Request) → POST /chat (IA Bloom)
  ─ Valida: mensaje (required)
  ─ Busca o crea conversación con type='ai' para el usuario
  ─ Guarda mensaje del usuario en messages_ai (role: 'user')
  ─ Llama a PsicologoService::generarRespuesta(historial, mensaje)
  ─ Guarda respuesta de Bloom en messages_ai (role: 'assistant')
  ─ Retorna: { respuesta: texto_bloom, status: 'ok' }

  ── CommunityController.php ───────────────────────────────────────────────────

  followingMoods(Request) → GET /community/following-moods
  ─ Obtiene IDs de usuarios que sigo (followed_follower WHERE follower_id = auth)
  ─ Para cada uno: último registro en mood_registers (orderBy date DESC, limit 1)
  ─ Retorna: [{ id, name, username, image_id, mood: string|null, mood_date: string|null }]

  suggestedUsers(Request) → GET /community/suggested
  ─ Excluye: los que ya sigo + yo mismo
  ─ Filtra: is_public = true
  ─ Ordena: por streak DESC, limit 10
  ─ Retorna: [{ id, name, username, image_id, streak, points }]

  getPosts(Request) → GET /community/posts?page=1
  ─ JOIN publications + users
  ─ Paginación: 20 por página, offset calculado
  ─ Para cada post: likesCount (COUNT post_likes), isLiked (EXISTS para auth user)
  ─ Retorna: [{ id, title, content, date, user_id, user_name, username, image_id, likes_count, is_liked }]

  createPost(Request) → POST /community/posts
  ─ Valida: content (required, max:1000)
  ─ INSERT en publications: { title:'', text_publications: content, date: now(), user_id: auth }
  ─ Hace JOIN para devolver datos completos del post recién creado
  ─ Retorna 201: { id, title, content, date, user_id, user_name, username, image_id, likes_count: 0, is_liked: false }

  likePost(Request, $id) → POST /community/posts/{id}/like
  ─ Comprueba si existe (user_id, publication_id) en post_likes
  ─ Si existe: DELETE → liked = false
  ─ Si no existe: INSERT → liked = true
  ─ Cuenta total de likes del post
  ─ Retorna: { liked: bool, likes_count: int }

  getStatus(Request) → GET /getStatus

  ─ Retorna: { exists: bool, data: { total_score, breakdown }|null }

  store(Request) → POST /store

  ─ Retorna: { status: 'success' }

  11. BACKEND — SERVICIO DE IA (PsicologoService.php)
================================================================================

  Ubicación: moodify_backend/app/Services/PsicologoService.php

  Función: Integración con Ollama para generar respuestas de Bloom (psicóloga IA)

  Modelo usado: qwen3-vl:30b (multimodal)
  Endpoint Ollama: POST {OLLAMA_HOST}/api/chat

  System prompt de Bloom:
  ─ Se presenta como psicóloga empática llamada Bloom
  ─ Tiene escucha activa, nunca juzga
  ─ Hace preguntas abiertas para profundizar
  ─ Responde siempre en el idioma del usuario
  ─ Tono cálido, cercano, profesional

  Método principal: generarRespuesta(array $historial, string $nuevoMensaje)
  ─ Formatea el historial de messages_ai como array de mensajes OpenAI-compatible
  ─ Añade el nuevo mensaje al final
  ─ Llama a Ollama con el historial completo (conversación continua)
  ─ Retorna: string con la respuesta de Bloom
  ─ En caso de error: retorna mensaje de fallback predefinido

  Integración en ChatController@store:
  1. Guarda mensaje del usuario → messages_ai (role: 'user')
  2. Recupera historial completo de la conversación
  3. Llama PsicologoService::generarRespuesta(historial, mensaje)
  4. Guarda respuesta → messages_ai (role: 'assistant')
  5. Retorna { respuesta, status: 'ok' } al frontend

  15. CONFIGURACIÓN DE ENTORNO
================================================================================

  ── FRONTEND (moodify-app/.env) ───────────────────────────────────────────────

  EXPO_PUBLIC_API_URL=http://localhost:8000/api/

  Notas:
  ─ Para emulador Android: cambiar a http://10.0.2.2:8000/api/
  ─ Para dispositivo físico: cambiar a http://{IP_LOCAL}:8000/api/
  ─ Requiere reinicio de Expo tras cambiar: pulsa 'r' en la terminal
  ─ El prefijo EXPO_PUBLIC_ es obligatorio para que Expo lo inyecte en el bundle

  ── BACKEND (moodify_backend/.env) ────────────────────────────────────────────

  APP_URL=http://localhost
  DB_CONNECTION=sqlite
  DB_DATABASE=moodify          ← Apunta al archivo ./moodify (sin extensión)
  OLLAMA_HOST="ipollama"       ← Host del servidor Ollama para la IA

  Notas:
  ─ El archivo de BD real es: moodify_backend/moodify (172KB aprox.)
  ─ moodify_backend/database/database.sqlite existe pero está VACÍO (0 bytes)
  ─ El servidor Laravel se inicia con: php artisan serve (puerto 8000 por defecto)
  ─ Sanctum ya está configurado para APIs (stateless, sin CSRF para API routes)

  ── COMANDOS ÚTILES ───────────────────────────────────────────────────────────

  # Backend
  cd moodify_backend
  php artisan serve                          → Inicia servidor en localhost:8000
  php artisan migrate:status                 → Ver estado de migraciones
  php artisan migrate                        → Ejecutar migraciones pendientes
  php artisan route:list                     → Ver todas las rutas
  sqlite3 moodify ".tables"                  → Ver tablas en la BD
  sqlite3 moodify "SELECT * FROM users;"     → Consultar datos

  # Frontend
  cd moodify-app
  npx expo start                             → Inicia Expo (web en localhost:8081)
  npx expo start --web                       → Fuerza modo web
  npx tsc --noEmit                           → Verificar TypeScript sin compilar
